在 Linux 中，评估一个进程“真实”占用的物理内存，**RES (Resident Set Size)** 通常是最接近的指标。

*   **RES (常驻内存): 2573M (约 2.5 GB)**
    *   这是该进程实际占用的物理内存大小，不包括被交换出去（Swap）的内存。
    *   截图中的多个 `p2p_proxy` 线程（或子进程）都显示相同的 RES 值，这通常意味着它们共享同一块内存空间（如果是多线程程序），或者这是父进程的内存视图。考虑到 PID 851 是主进程（或者是线程组的代表），且 MEM% 都是 35.4%，可以认为该程序作为一个整体占用了大约 **2.57 GB** 的物理内存。

其他相关指标：
*   **VIRT (虚拟内存): 4180M (约 4.18 GB)**
    *   这是进程申请的所有虚拟地址空间的总和，包括了已映射但未使用的内存、共享库等，通常远大于实际使用的物理内存。
*   **SHR (共享内存): 13660 (约 13 MB)**
    *   这是与其他进程共享的内存大小（如共享库），这部分内存不完全属于该进程私有。
*   **MEM% (内存占比): 35.4%**
    *   该进程占用了系统总物理内存的 35.4%。

**结论：**
`p2p_proxy` 当前实际使用的物理内存（真实内存）约为 **2.57 GB**。

但是我已经设置了使用最大512M内存啊？为什么会这样

## Copilot

这是一个非常经典的问题，特别是在使用 Go 语言开发网络应用（如 libp2p）时。

简单来说：**`libp2p.ResourceManager` (rcmgr) 设置的是“逻辑预算”，而不是操作系统的“物理硬限制”。**

你虽然在代码里告诉 libp2p "只用 512MB"，但 `htop` 显示占用了 2.5GB (RES)，主要原因通常是以下几点：

### 1. Go 运行时的 GC 机制 (最主要原因)

Go 语言是带垃圾回收（GC）的。Go 的默认行为是：**“只要系统还有空闲内存，我就不急着把用完的内存归还给操作系统”。**

*   **现象**：你的程序可能实际上只“活跃”使用了 300MB 数据，但 Go 为了复用内存（避免反复向系统申请释放的性能开销），会持有大量的空闲内存（Idle Memory）。
*   **证据**：看你的 `htop`，系统有 7GB 内存，空闲很多。Go 运行时会认为“现在内存很富裕，没必要频繁 GC，也没必要把内存还给 OS”。
*   **解决方案**：
    从 Go 1.19 开始，官方引入了软内存限制。请在启动程序时添加环境变量：
    ```bash
    export GOMEMLIMIT=500MiB
    ./p2p_proxy_linux
    ```
    或者在 Systemd 服务文件中添加 `Environment="GOMEMLIMIT=500MiB"`。
    *这会强制 Go 的 GC 激进工作，尽量把内存压在 500MB 以内，超过时会强制回收，甚至返还给 OS。*

### 2. ResourceManager (rcmgr) 的统计偏差

`rcmgr` 并不是监控整个进程的内存，它只是在**记账**。

*   **它的工作方式**：当你打开一个流，rcmgr 记账 "+4KB"；当你建立连接，记账 "+X KB"。
*   **它的盲区**：
    *   **业务逻辑内存**：如果你在代码里 `make([]byte, 10MB)` 来处理业务数据，`rcmgr` 是不知道的，它管不了这部分。
    *   **Go 运行时开销**：Goroutine 的栈空间（每个起始 2KB，但会动态增长）、GC 的元数据、程序二进制本身的代码段，都不在 `rcmgr` 的 512MB 预算里。
    *   **Peerstore (节点存储)**：这是 libp2p 的一个常见内存泄露点。如果你连接过几万个节点，libp2p 会把这些节点的 ID、地址、Key 存在内存里的 `peerstore` 中，这部分内存**通常不包含**在你设置的 `rcmgr` 限制里。

### 3. Relay V2 的缓冲区与 Goroutine

你配置了 Relay：
```go
relayResources.BufferSize = 4096
relayResources.MaxCircuits = 1024
```
*   **Goroutine 数量**：每个活跃的 Relay 电路通常需要 2 个 goroutine（读/写方向各一个）。如果是 1000 个电路，就是 2000 个 goroutine。
*   **内存开销**：虽然你限制了 Buffer 是 4KB，但在高并发下，Go 的调度器、网络轮询器（Netpoller）以及每个 goroutine 的栈空间（如果递归深或者局部变量大，栈会扩容）都会消耗大量 `RES` 物理内存。

### 4. 你的代码逻辑建议

要解决这个问题，不仅要依赖代码里的 `rcmgr`，还需要配合 Go 运行时的调优。建议按以下步骤排查和优化：

#### 第一步：加上 GOMEMLIMIT (必须做)
这是最立竿见影的。
```bash
# 建议设置为你目标内存的 90% 左右，留一点余量给 OS
GOMEMLIMIT=450MiB ./p2p_proxy_linux
```

#### 第二步：优化 Peerstore (建议做)
如果你的程序运行久了内存才变大，大概率是 Peerstore 存了太多死节点的地址。
在创建 host 时，配置自动清理 Peerstore：

```go
// 引入 "github.com/libp2p/go-libp2p/p2p/host/peerstore/pstoreds" (如果用数据库)
// 或者在内存模式下，手动清理，或者使用 BasicTagManager

// 简单粗暴的方法：配置 ConnectionManager 激进一点
connmgr, _ := connmgr.NewConnManager(
    100, // Lowwater: 连接数低于这个值开始在这个范围内保持
    400, // Highwater: 连接数高于这个值开始踢人
    connmgr.WithGracePeriod(time.Minute),
)

h, err := libp2p.New(
    // ... 其他配置
    libp2p.ConnectionManager(connmgr), // 限制同时连接的物理连接数
)
```
*注意：`rcmgr` 限制的是资源总量，`connmgr` 负责主动断开多余的连接。两者需要配合。*

#### 第三步：使用 pprof 查看到底是谁占用了内存 (进阶)

如果加了 `GOMEMLIMIT` 还是很占内存（或者 OOM 崩溃），你需要看看到底是哪部分代码在吃内存。

在你的 `main` 函数最开始加上：

```go
import (
    "net/http"
    _ "net/http/pprof"
)

func main() {
    // 开启一个 debug 端口
    go func() {
        log.Println(http.ListenAndServe("0.0.0.0:6060", nil))
    }()
    
    // ... 原有逻辑
}
```

程序运行起来后，在本地电脑终端执行：
```bash
go tool pprof -http=:8080 http://<你的服务器IP>:6060/debug/pprof/heap
```
这会打开一个网页，直观地告诉你到底是 `relay` 的 buffer 占用了内存，还是 `peerstore`，或者是其他的业务逻辑。

### 总结
你现在的 2.5GB 占用，主要是 **Go 运行时偷懒没归还内存** + **libp2p 的隐式开销（Goroutine栈/Peerstore）**。

请优先设置环境变量 `GOMEMLIMIT=480MiB`，通常能立刻将内存压回到 500MB 左右的水平。

## Copilot

你需要将 `Environment="GOMEMLIMIT=500MiB"` 添加到 `p2p_proxy.service` 文件中的 **`[Service]`** 部分。

具体操作步骤如下：

### 1. 修改 Service 文件

通常该文件位于 `/etc/systemd/system/p2p_proxy.service` (或者 `/usr/lib/systemd/system/`，取决于你的安装方式)。

在该文件中找到 `[Service]` 这一行，将环境变量加在它下面的任意位置（通常在 `ExecStart` 附近）。

**示例配置：**

```ini
[Unit]
Description=P2P Proxy Service
After=network.target

[Service]
Type=simple
User=root
# 你的启动命令
ExecStart=/www/server/p2p/bin/p2p_proxy_linux

# ✅ 在这里添加环境变量
Environment="GOMEMLIMIT=500MiB"

# 如果你想更激进一点，可以配合 GOGC (默认是 100)
# Environment="GOGC=50"

Restart=on-failure

[Install]
WantedBy=multi-user.target
```

### 2. 生效配置

修改完成后，必须执行以下两个命令才能生效：

```bash
# 1. 重载系统配置
sudo systemctl daemon-reload

# 2. 重启你的服务
sudo systemctl restart p2p_proxy
```

### 💡 为什么需要加这个？（关键补充）

你在代码中写的 `rcmgr` (Resource Manager) 限制：
```go
var memoryLimit int64 = 512 * 1024 * 1024
```
这**只限制了 libp2p 组件**（如连接数、流缓冲区）能申请的内存。它**不包括**：
1.  Go 语言运行时的开销（Runtime overhead）。
2.  代码中其他非 libp2p 部分申请的内存。
3.  Go 垃圾回收（GC）还没来得及回收的“惰性”内存。

**`GOMEMLIMIT` 的作用是：**
它直接告诉 Go 的垃圾回收器：“当内存使用量接近 500MiB 时，**强制**进行垃圾回收，不要再为了性能而推迟回收了。” 这对于防止内存泄漏或过度占用非常有效。

那在windows 2012和windows 7应该如何设置GOMEMLIMIT？

## Copilot

在 Windows Server 2012 和 Windows 7 上设置环境变量 `GOMEMLIMIT` 主要有三种方式。你可以根据你是如何启动程序的（手动启动、通过脚本启动、还是作为后台服务运行）来选择最合适的方法。

以下是具体操作步骤：

### 方法一：通过 CMD 或 PowerShell 启动（临时生效）

如果你是在命令行窗口中手动运行程序，可以在运行前设置变量。这只对当前窗口有效，关闭窗口后失效。

**CMD (命令提示符):**
```cmd
set GOMEMLIMIT=500MiB
p2p_proxy.exe
```

**PowerShell:**
```powershell
$env:GOMEMLIMIT="500MiB"
.\p2p_proxy.exe
```

---

### 方法二：创建启动脚本（推荐用于手动运行）

为了方便，你可以创建一个 `.bat` 批处理文件来启动你的程序。这样每次只需要双击这个文件即可。

1.  新建一个文本文档，重命名为 `start_proxy.bat`。
2.  右键点击 -> **编辑**。
3.  输入以下内容（假设 `p2p_proxy.exe` 在同一目录下）：

```bat
@echo off
:: 设置 Go 内存限制
set GOMEMLIMIT=500MiB

:: 启动程序
echo Starting p2p_proxy with GOMEMLIMIT=%GOMEMLIMIT%...
p2p_proxy.exe

:: 如果程序崩溃或关闭，暂停一下以便查看报错
pause
```
4.  保存并关闭。以后双击这个 `.bat` 文件运行即可。

---

### 方法三：设置为系统全局环境变量（永久生效）

如果你希望无论怎么启动该程序（包括双击 exe），它都能读取到这个限制，或者你是用 NSSM 等工具将其注册为服务，建议设置系统环境变量。

1.  **打开环境变量设置界面**：
    *   **Windows 7 / 2012**: 右键点击桌面的 **"计算机" (Computer)** -> **"属性" (Properties)** -> 左侧点击 **"高级系统设置" (Advanced system settings)**。
    *   在弹出的对话框中，点击 **"环境变量" (Environment Variables)** 按钮。

2.  **新建系统变量**：
    *   在下方的 **"系统变量" (System variables)** 区域（注意不是上面的“用户变量”），点击 **"新建" (New)**。
    *   **变量名 (Variable name)**: `GOMEMLIMIT`
    *   **变量值 (Variable value)**: `500MiB`

3.  **保存并重启**：
    *   点击 **确定** 保存所有设置。
    *   **重要**：你需要**重启你的程序**，甚至可能需要**重启电脑**（如果程序是作为开机自启的服务运行的），设置才会生效。

---

### 方法四：如果作为 Windows 服务运行 (例如使用 NSSM)

如果你使用 **NSSM** (Non-Sucking Service Manager) 将程序注册为了 Windows 服务，你需要专门为该服务设置环境变量。

1.  以管理员身份打开 CMD。
2.  运行以下命令（将 `YourServiceName` 替换为你的服务名称）：

```cmd
nssm set YourServiceName AppEnvironmentExtra GOMEMLIMIT=500MiB
```

3.  重启服务：
```cmd
nssm restart YourServiceName
```

### 验证是否生效

你可以通过任务管理器观察。设置生效后，当内存占用接近 500MB 时，Go 应该会开始频繁进行垃圾回收，内存曲线会趋于平稳，不再像之前那样无限制增长。

## Copilot

可以。你当前用的是 pywin32 的 ShellExecute 启动 exe，有几种把 GOMEMLIMIT 传给子进程的做法，按从简单到可控排序给出，并附示例（均为 Python 2.7 可用的写法）。

要点总结
- 子进程会继承父进程（也就是你运行 Python 脚本的进程）的环境变量。只要在调用启动前把环境设置好，子进程就能看到 GOMEMLIMIT。
- Windows 全局/永久环境变量需要改注册表并广播设置变化；通常不需要这么做，只需在启动时为该进程注入即可。
- 推荐使用 subprocess.Popen（可显式传 env），若想继续用 ShellExecute 可直接修改 os.environ 或用 win32api.SetEnvironmentVariable。

方法 A：最简单 — 在 Python 进程里设置环境，然后用 ShellExecute（适合你当前代码）
```python
import os
import win32api

# 设置当前 Python 进程的环境变量（子进程会继承）
os.environ['GOMEMLIMIT'] = '500MiB'
# 或者使用 pywin32 接口（效果类似，只对当前进程）
# win32api.SetEnvironmentVariable('GOMEMLIMIT', '500MiB')

# 之后再调用 ShellExecute 启动 exe
win32api.ShellExecute(0, '', r'C:\path\to\p2p_proxy.exe', '', r'C:\path\to', 0)
```
说明：在调用 ShellExecute 之前把 os.environ 设置好即可。注意这只影响当前 Python 进程及其子进程（不会修改系统全局环境）。

方法 B：更可靠／推荐 — 使用 subprocess.Popen，显式传 env（适用于控制工作目录、捕获输出等）
```python
import os
import subprocess

env = os.environ.copy()
env['GOMEMLIMIT'] = '500MiB'

p = subprocess.Popen(
    [r'C:\path\to\p2p_proxy.exe'],
    cwd=r'C:\path\to',
    env=env,
    shell=False
)
# 如果想等它退出，使用 p.wait()
```
说明：显式传 env 最稳妥，不依赖全局或父进程的环境修改。

方法 C：用 pywin32 的 CreateProcess，显式传入 environment block（更底层、适合需要 Win32 控制的场景）
```python
import os
import win32process
import win32con

cmd = r'C:\path\to\p2p_proxy.exe'
cwd = r'C:\path\to'
env = os.environ.copy()
env['GOMEMLIMIT'] = '500MiB'

# win32process.CreateProcess 支持 dict 或字符串环境
pi = win32process.CreateProcess(
    None,          # application name
    cmd,           # command line
    None, None,    # procAttr, threadAttr
    False,         # bInheritHandles
    0,             # dwCreationFlags
    env,           # environment: dict 或者 "A=B\0C=D\0\0" 形式
    cwd,           # current directory
    None           # STARTUPINFO (使用默认)
)
```

方法 D：如果你想永久地在系统中添加（不推荐，仅说明）
- 修改系统环境变量（控制面板 → 系统 → 高级系统设置 → 环境变量），或通过注册表写入，然后广播 WM_SETTINGCHANGE，这样系统中所有新进程都会看到。但这会影响整台机器，不建议仅为单一程序使用。

如何验证 GOMEMLIMIT 是否生效
- 使用 Process Explorer（Sysinternals）打开该进程，右键 → Properties → Environment，查看 GOMEMLIMIT 是否存在。
- 或者在程序里临时打印 os.Getenv("GOMEMLIMIT")（如果你能修改 p2p_proxy 的代码）。

关于值格式
- Go 支持类似 "500MiB"、"480MiB"、"1GiB" 的单位写法，建议使用 MiB/GiB 格式，例如 "500MiB"。

小结和建议
- 最快速的做法：在你启动 p2p_proxy 之前在 Python 里 set os.environ，然后调用 ShellExecute（方法 A）。
- 更稳妥和可控：改用 subprocess.Popen，显式传入 env（方法 B）。
- 启动后用 Process Explorer 或 p2p_proxy 自己输出确认环境变量确实生效。

那gov1.10没GOMEMLIMIT，应该如何限制内存的使用？

## Copilot

Go 1.10 没有 GOMEMLIMIT（这是 1.19+ 的特性），所以不能靠运行时内部的“软限制”来强制 GC。要把进程内存严格控制在一个上限，有两类方案可以并用：

- 应用/语言层面的“减肥与更积极的 GC”策略（对旧版 Go 可用，降低内存增长、更频繁回收）
- 操作系统层面的强制限制（更可靠，超过限制会被 OS 限制或杀掉）

下面按可操作性和效果给出具体办法与示例。

1) 在 Go 程序里做能做的事（Go1.10 可用）
- 降低 GC 灵敏度：设置 GOGC（环境变量）或在代码里用 runtime/debug.SetGCPercent 调整 GC 阈值，使 GC 更频繁，保持堆更小。
  - 环境变量（启动前设）：
    - export GOGC=50  （默认 100，越小 GC 越激进，内存越低但 CPU 消耗更高）
  - 代码内设置（Go1.10 支持）：
    ```go
    import "runtime/debug"

    func init() {
        // 设为 20~50 之间的值，视 CPU/内存权衡调整
        debug.SetGCPercent(50)
    }
    ```
- 主动触发 GC 并尝试归还内存：
  - runtime.GC() 会触发 GC； debug.FreeOSMemory() 会触发 GC 并尽力把可释放内存还给操作系统（但在旧版本上效果有限，且不是保证会全部返还）。
  - 可以在低峰期用 ticker 周期性触发：
    ```go
    import (
        "runtime"
        "runtime/debug"
        "time"
    )

        t := time.NewTicker(30 * time.Second)
        for range t.C {
            runtime.GC()
            debug.FreeOSMemory()
        }
    }()
    ```
  注意：这些方法只是“更积极地回收”，无法完全保证物理 RES 不超出目标，且会增加 CPU。

- 减少内存使用的编码/库配置
  - 限制 buffer/queue/stream 大小（你已经在 relay 配置里做了）。
  - 尽量避免在内存中保持大量 peer info、消息缓存等（使用磁盘/数据库或 TTL 清理）。
  - 限制 goroutine 与栈增长（避免在热路径创建大量短生命周期但栈会增长的 goroutine）。

2) 在 Linux 上：使用 OS 层面的强制限制（最可靠）
- systemd 服务：在 service 文件的 [Service] 部分加：

  ```

    ```
    prlimit --as=536870912 --pid <pid>
    ```

    ```
    ulimit -v 524288   # 512*1024 KB = 524288

    ```

func startGCMonitor() {
	ticker := time.NewTicker(time.Duration(gcCheckSeconds) * time.Second)
	defer ticker.Stop()

	var lastTriggered time.Time
	minIntervalBetweenTriggers := time.Duration(gcCheckSeconds) * time.Second / 2
	if minIntervalBetweenTriggers < 5*time.Second {
		minIntervalBetweenTriggers = 5 * time.Second
	}

	for range ticker.C {
		var ms runtime.MemStats
		runtime.ReadMemStats(&ms)
		alloc := int64(ms.Alloc) // 当前实际占用堆（字节）
		sys := int64(ms.Sys)     // 分配给运行时的总字节

		// 记录基本状态（可视化或上报时取消注释）
		log.Printf("ℹ️ Mem: Alloc=%d KB, Sys=%d KB, HeapObjects=%d",

		// 如果超过软上限，或占比非常高，则触发更积极回收

		if shouldTrigger && time.Since(lastTriggered) > minIntervalBetweenTriggers {

说明与调参建议（中文，针对 Go 1.10）：

- 为什么需要这样做：
  - Go 1.10 没有 GOMEMLIMIT，只有 GOGC/SetGCPercent 可以控制垃圾回收触发的阈值。更激进的 GC 会更频繁回收堆，减少堆峰值，但会增加 CPU 使用。
  - debug.FreeOSMemory() 会在 GC 后尝试把空闲页面返还给操作系统，但在不同平台/Go 版本上效果不同（有时并不能完全返还）。
  - 因此我们使用：1) 设置较低的 GCPercent（更频繁触发 GC）；2) 定期检查当前 Alloc，当接近目标软上限时强制触发 runtime.GC() 并 optionally 调用 debug.FreeOSMemory()。

- GOGC / debug.SetGCPercent 的推荐范围（经验值）：
  - 默认 GOGC=100：性能与内存折中，适合多数后台服务。
  - 如果目标是更小内存（例如 512MB 上线），建议设置为 20 ~ 60 范围：
    - 20~30：非常激进，内存占用低但 CPU 消耗明显上升。适合内存非常紧张的场景或少量并发。
    - 40~60：折中，通常能显著降低内存峰值而 CPU 增幅可接受。
    - 70~100：保守，适合对延迟/CPU 更敏感的服务。
  - 示例：对 512MB 目标，首选尝试 GOGC=50（SetGCPercent(50)）；如果 CPU 可以承受，可下调到 30；若影响太大再上调。

- MEM_SOFT_LIMIT_MB（示例代码中的 memSoftLimit）
  - 这是一个“触发更激进回收”的阈值。建议设为���期望进程在正常运行下的上限（例如你想限制在 500MB，则设 480MB 作为软上限）。
  - 当 Alloc 超过该值的 95%（或 120%）时，会触发一次强制 GC + FreeOSMemory。

- 周期检查间隔（GC_CHECK_SECONDS）
  - 10~60 秒之间通常合适。太短会频繁读取 stats 与触发 GC，增加开销。太长意味着内存可能在阈值上滞留较久。
  - 示例代码默认 30 秒。

- 监控与验证
  - 在生产环境试验时，先监控 CPU 与内存曲线：
    - 观察 htop / top 的 RES 与 VIRT。
    - 使用 pprof（在程序中暴露 /debug/pprof）抓堆快照，查看哪些对象占内存。
  - 在调整 GOGC 时注意观察 95/99 百分位延迟和 CPU 使用率，找平衡点。

- 补充说明
  - 这些技术可以显著降低内存峰值，但无法像 cgroup / systemd MemoryLimit 或 Windows Job Object 那样做“硬限制”。如果希望强制在 OS 级别断开/杀死超限进程，仍需使用 OS 的资源控制（Linux cgroup、systemd MemoryLimit、Windows Job Object / container）。

// dialWithFallback 先尝试 ipv4，再尝试默认协议；使用 net.Dialer 可查看本端地址
func dialWithFallback(addr string, timeout time.Duration) (net.Conn, error) {
	d := &net.Dialer{Timeout: timeout, KeepAlive: 30 * time.Second}
	// 优先尝试 IPv4（在老系统上有时更可靠）
	conn, err := d.Dial("tcp4", addr)
	if err == nil {
		log.Printf("✅ Dial tcp4 成功，本地地址: %v -> %v", conn.LocalAddr(), conn.RemoteAddr())
		return conn, nil
	}
	// 记录第一次错误，便于诊断
	log.Printf("⚠️ tcp4 失败: %v，尝试默认 tcp...", err)
	conn, err = d.Dial("tcp", addr)
	if err == nil {
		log.Printf("✅ Dial tcp 成功，本地地址: %v -> %v", conn.LocalAddr(), conn.RemoteAddr())
	}
	return conn, err
}

// 新增：判断是否为本机/回环地址
func isLocalIPAddress(ip string) bool {
	if ip == "" || ip == "127.0.0.1" || strings.EqualFold(ip, "localhost") {
		return true
	}
	parsed := net.ParseIP(ip)
	if parsed == nil {
		return false
	}

	addrs, err := net.InterfaceAddrs()
	if err != nil {
		return false
	}
	for _, a := range addrs {
		switch v := a.(type) {
		case *net.IPNet:
			if v.IP.Equal(parsed) {
				return true
			}
		case *net.IPAddr:
			if v.IP.Equal(parsed) {
				return true
			}
		}
	}
	return false
}

// 新增函数：获取第一个非回环 IPv4 地址（找不到时返回 127.0.0.1）
func getFirstLocalIPv4() string {
	interfaces, err := net.Interfaces()
	if err != nil {
		return "127.0.0.1"
	}
	for _, iface := range interfaces {
		// 跳过不可用或回环接口
		if iface.Flags&net.FlagUp == 0 || iface.Flags&net.FlagLoopback != 0 {
			continue
		}
		addrs, err := iface.Addrs()
		if err != nil {
			continue
		}
		for _, a := range addrs {
			var ip net.IP
			switch v := a.(type) {
			case *net.IPNet:
				ip = v.IP
			case *net.IPAddr:
				ip = v.IP
			}
			if ip == nil || ip.IsLoopback() {
				continue
			}
			if ipv4 := ip.To4(); ipv4 != nil {
				return ipv4.String()
			}
		}
	}
	return "127.0.0.1"
}

// getAllLocalIPs 返回本机所有非回环、非链路本地的 IP 地址，使用分号分隔。
// 若未找到可用地址，返回 "127.0.0.1"。
func getAllLocalIPs() string {
	interfaces, err := net.Interfaces()
	if err != nil {
		return "127.0.0.1"
	}
	var parts []string
	for _, iface := range interfaces {
		// 跳过不可用或回环接口
		if iface.Flags&net.FlagUp == 0 || iface.Flags&net.FlagLoopback != 0 {
			continue
		}
		addrs, err := iface.Addrs()
		if err != nil {
			continue
		}
		for _, a := range addrs {
			var ip net.IP
			switch v := a.(type) {
			case *net.IPNet:
				ip = v.IP
			case *net.IPAddr:
				ip = v.IP
			}
			if ip == nil || ip.IsLoopback() {
				continue
			}
			// 过滤链路本地地址：IPv4 169.254.x.x，IPv6 fe80::
			if ipv4 := ip.To4(); ipv4 != nil {
				if ipv4[0] == 169 && ipv4[1] == 254 {
					continue
				}
				parts = append(parts, ipv4.String())
			} else {
				s := ip.String()
				if strings.HasPrefix(strings.ToLower(s), "fe80:") {
					continue
				}
				parts = append(parts, s)
			}
		}
	}
	if len(parts) == 0 {
		return "127.0.0.1"
	}
	return strings.Join(parts, ";")
}
